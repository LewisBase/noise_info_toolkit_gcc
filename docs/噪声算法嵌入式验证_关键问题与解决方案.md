# **噪声算法嵌入式验证：关键问题与解决方案**

根据上传文件《算法测试结果.docx》的内容整理。本文完整保留上一条答复中的原文字、表格、公式式样和代码块，并转换为 Word 格式。

# **一、关键问题在哪里？**

我仔细阅读了上传的《算法测试结果.docx》。这份文件的核心信息很明确：**当前噪声算法已经能够在嵌入式环境中进入编译/运行测试阶段，但暴露出三个关键问题：编译兼容性、堆内存随数据量增长、实时处理时间接近采集时间预算。** 其中真正严重的不是 M\_PI 未定义，而是后两项：**算法仍然采用 PC 端批处理式 vector 结构，不适合 nRF54L15 这类 256 KB SRAM 的实时嵌入式平台。** 文件中明确指出，当前算法在 process\_segment、IIRFilter::process、倍频带分支等位置会反复构造 std::vector<float>、复制整段数据，并通过 operator new 落到 Zephyr 的 k\_heap，导致堆占用随段长增加而增加；同时，处理 2400 个样本的耗时已经与 48 kHz 下采满 2400 个样本的 50 ms 同量级，存在实时积压或丢数风险。

## **关键问题 1：\`M\_PI\` 未定义只是表层兼容问题**

文件中第一条提到，在 Zephyr / arm-zephyr-eabi 下使用 C++17 编译算法时，<cmath> 不一定提供 M\_PI，直接写 M\_PI 会报错。

这不是算法本身的问题，而是**跨平台 C/C++ 标准库兼容性问题**。

M\_PI 并不是 C++ 标准正式保证的常量。它在很多桌面系统里可用，是因为某些库额外提供了这个宏；但在 Zephyr、picolibc、arm-zephyr-eabi 这种嵌入式工具链下，不能假设它一定存在。

### **解决方案**

不要依赖 M\_PI。在公共头文件中定义自己的常量：

<table><tr><td><p>#pragma once</p><p> </p><p>namespace noise_const {</p><p>    inline constexpr float PI_F =</p><p>        3.14159265358979323846f;</p><p> </p><p>    inline constexpr float TWO_PI_F =</p><p>        2.0f * PI_F;</p><p>}</p></td></tr></table>

所有嵌入式实时算法使用：

<table><tr><td><p>float omega = noise_const::TWO_PI_F * freq / sample_rate;</p></td></tr></table>

而不是：

<table><tr><td><p>float omega = 2.0f * M_PI * freq / sample_rate;</p></td></tr></table>

同时要注意，常量后面应加 f，否则 3.14159265358979323846 默认是 double，可能引发隐式 double 运算。

正确写法：

<table><tr><td><p>constexpr float PI_F = 3.14159265358979323846f;</p></td></tr></table>

不推荐：

<table><tr><td><p>#define M_PI 3.14159265358979323846</p></td></tr></table>

因为宏没有类型控制，也容易污染命名空间。

## **关键问题 2：堆内存随段长增长，这是主要风险**

文件指出，算法在处理每一段数据时，会在运行期向堆申请临时空间；单段样本越多、数据量越大，单次处理的堆占用峰值越高，并且这些分配主要来自 std::vector。

具体位置包括：

1. noise\_processor.cpp 的 process\_segment：为 A/C 计权各构造一次 std::vector<float>(buffer\_start, buffer\_end)，也就是把整段音频复制一遍。

2. apply\_a\_weighting / apply\_c\_weighting 内部：多级滤波过程中继续使用 std::vector 和输出缓冲。

3. iir\_filter.cpp 的 IIRFilter::process / BiquadFilter::process：为输出构造 std::vector<float> result，并按样本 push\_back。

4. 9 个倍频带分支：每个频带又重复：拷贝输入段 → 构造滤波器 → 输出 vector → 计算结果。

这说明当前实现还是典型 PC 端算法写法：

<table><tr><td><p>输入一整段 vector</p><p>↓</p><p>复制 vector</p><p>↓</p><p>滤波后产生新的 vector</p><p>↓</p><p>再复制到下一级</p><p>↓</p><p>最后统计</p></td></tr></table>

这种结构在 PC 上问题不大，但在 nRF54L15 上非常危险。

nRF54L15 只有 256 KB SRAM。Zephyr 协议栈、线程栈、驱动、BLE、文件系统、日志缓冲、DMA buffer 都要占用 RAM。如果算法热路径再反复申请大块 vector，就会出现：

<table><tr><td><p>段长越大 → vector 越大 → heap 峰值越高</p><p>↓</p><p>长期运行 → heap 碎片化</p><p>↓</p><p>偶发分配失败 / 处理超时 / 设备 hang</p></td></tr></table>

### **关键判断**

这个问题**不能靠单纯增大 \`CONFIG\_HEAP\_MEM\_POOL\_SIZE\` 解决**。

增大 heap 只能让当前测试暂时通过，但不是产品级解决方案。因为：

1. 256 KB SRAM 总量有限，heap 增大就会挤压 stack、BLE、DMA、文件系统；

2. heap 峰值随段长和倍频带数量变化，不稳定；

3. 长时间运行后可能碎片化；

4. 实时采集系统不能接受不可预测的 malloc/free 延迟；

5. 将来做 IEC/ANSI 类设备验证时，固定内存预算比动态堆更容易审计。

### **正确解决方向**

**嵌入式热路径必须做到零堆分配。**

也就是：

<table><tr><td><p>process_segment() 内不能 new</p><p>process_segment() 内不能 malloc</p><p>process_segment() 内不能构造临时 std::vector</p><p>filter.process() 不能返回 std::vector</p><p>每个频带不能复制整段输入</p></td></tr></table>

## **关键问题 3：处理时间与采集时间同量级，实时裕量不足**

文件第三条指出，在固定算力和算法路径下，一段输入在 CPU 上跑完的时间，与按采样率采满同样样本数所需的时间相当；例如 48 kHz 下处理 2400 个样本的一帧，其处理耗时与 50 ms 墙钟同量级。

这说明当前算法已经接近实时边界。

48 kHz 下：

<table><tr><td><p><strong>段长</strong></p></td><td><p><strong>样本数</strong></p></td><td><p><strong>采集时间预算</strong></p></td></tr><tr><td><p>10 ms</p></td><td><p>480 samples</p></td><td><p>10 ms</p></td></tr><tr><td><p>20 ms</p></td><td><p>960 samples</p></td><td><p>20 ms</p></td></tr><tr><td><p>50 ms</p></td><td><p>2400 samples</p></td><td><p>50 ms</p></td></tr><tr><td><p>100 ms</p></td><td><p>4800 samples</p></td><td><p>100 ms</p></td></tr></table>

如果处理 2400 个样本需要接近 50 ms，那么系统没有足够裕量做：

I2S/PDM 中断处理；

BLE 通信；

外部 Flash 写入；

OLED 刷新；

按键/报警；

文件系统；

电池管理；

低功耗调度；

事件录音；

异常处理。

而且当前处理时间中包含大量 vector 分配、复制和释放。也就是说，算法实际数学计算未必是唯一瓶颈，**内存分配和数据搬运本身就消耗了大量时间**。

### **实时系统的目标不应是“刚好算完”**

如果采集 50 ms，处理也要 50 ms，这是不安全的。

比较合理的目标是：

<table><tr><td><p>处理时间 ≤ 采集时间的 30%–50%</p></td></tr></table>

例如：

<table><tr><td><p><strong>段长</strong></p></td><td><p><strong>采集时间</strong></p></td><td><p><strong>建议处理时间目标</strong></p></td></tr><tr><td><p>10 ms</p></td><td><p>10 ms</p></td><td><p>≤ 3–5 ms</p></td></tr><tr><td><p>20 ms</p></td><td><p>20 ms</p></td><td><p>≤ 6–10 ms</p></td></tr><tr><td><p>50 ms</p></td><td><p>50 ms</p></td><td><p>≤ 15–25 ms</p></td></tr></table>

这样系统才有余量处理通信、存储、显示和突发任务。

# **二、当前算法的根本问题**

这份测试结果暴露的根本问题可以概括为一句话：

*当前算法还是“桌面端批处理算法”，不是“嵌入式流式实时算法”。*

当前逻辑大概是：

<table><tr><td><p>一段音频进来</p><p>↓</p><p>复制成 vector</p><p>↓</p><p>A 计权处理，生成 A-weighted vector</p><p>↓</p><p>C 计权处理，生成 C-weighted vector</p><p>↓</p><p>多个倍频带分别复制和滤波</p><p>↓</p><p>再对各 vector 计算指标</p></td></tr></table>

这种设计有三个后果：

1. **RAM 高**：因为每个处理环节都可能生成一个与输入段等长的临时数组。

2. **CPU 慢**：因为存在大量内存拷贝、push\_back、动态分配、释放。

3. **实时性不可预测**：因为 heap 分配耗时不是固定的，长时间运行后还可能碎片化。

对个人噪声剂量计来说，正确思路应该是：

<table><tr><td><p>样本或小块输入</p><p>↓</p><p>直接进入 A/C/Z 滤波器</p><p>↓</p><p>滤波器只保存状态，不保存整段输出</p><p>↓</p><p>每个样本立即更新能量、峰值、峰度、剂量累计</p><p>↓</p><p>每秒输出一次 SecondMetrics</p></td></tr></table>

也就是：

*不保存中间波形，只保存状态和统计量。*

# **三、推荐的总体解决方案**

我建议分成三个阶段处理。

## **阶段 0：立即修复编译与测试阻塞问题**

这个阶段目标是：**让工程测试继续推进，但不把临时方案当成最终产品方案。**

### **0.1 修复 \`M\_PI\`**

在公共头文件中加入：

<table><tr><td><p>namespace noise_const {</p><p>    inline constexpr float PI_F =</p><p>        3.14159265358979323846f;</p><p> </p><p>    inline constexpr float TWO_PI_F =</p><p>        2.0f * PI_F;</p><p>}</p></td></tr></table>

替换所有：

<table><tr><td><p>M_PI</p></td></tr></table>

为：

<table><tr><td><p>noise_const::PI_F</p></td></tr></table>

同时把所有数学函数改成 float 版本：

<table><tr><td><p>log10f()</p><p>sqrtf()</p><p>fabsf()</p><p>powf()</p><p>sinf()</p><p>cosf()</p></td></tr></table>

避免：

<table><tr><td><p>log10()</p><p>sqrt()</p><p>fabs()</p><p>pow()</p><p>sin()</p><p>cos()</p></td></tr></table>

### **0.2 临时增大 heap 只用于验证**

可以临时设置：

<table><tr><td><p>CONFIG_HEAP_MEM_POOL_SIZE=65536</p></td></tr></table>

或更高，用于确认算法功能是否正确。

但必须明确：

*这是测试过渡方案，不是产品方案。*

最终嵌入式热路径应做到：

<table><tr><td><p>运行期 heap allocation = 0</p></td></tr></table>

### **0.3 暂时关闭倍频带分支**

如果当前 9 个倍频带分支显著增加堆和耗时，建议先定义：

<table><tr><td><p>NOISE_ALGO_SKIP_THIRD_OCTAVE</p></td></tr></table>

或类似编译选项。

先验证核心剂量计指标：

<table><tr><td><p>LAeq</p><p>LCeq</p><p>LZeq</p><p>Lmax</p><p>LCpeak</p><p>Dose</p><p>TWA / LEX,8h</p><p>crest factor</p><p>kurtosis</p></td></tr></table>

等核心路径稳定后，再逐步恢复倍频带分析。

## **阶段 1：把 vector 批处理改为无堆流式处理**

这是最关键的一步。

### **1.1 修改 \`process\_segment\` 接口**

当前问题是 process\_segment 内部把输入拷贝成 std::vector<float>。

建议改为指针 + 长度接口：

<table><tr><td><p>struct SampleBlock {</p><p>    const int32_t* samples;</p><p>    size_t count;</p><p>};</p><p> </p><p>void process_block(const SampleBlock&amp; block);</p></td></tr></table>

或者：

<table><tr><td><p>void process_block(const int32_t* samples, size_t count);</p></td></tr></table>

如果输入已经是 float，也可以：

<table><tr><td><p>void process_block(const float* samples, size_t count);</p></td></tr></table>

但不应在函数内部构造：

<table><tr><td><p>std::vector&lt;float&gt;(buffer_start, buffer_end)</p></td></tr></table>

### **1.2 滤波器改成逐样本处理**

当前 IIRFilter::process 返回 std::vector<float> result，这是堆分配来源之一。

应改成：

<table><tr><td><p>class BiquadFilter {</p><p>public:</p><p>    float process_sample(float x) {</p><p>        float y = b0 * x + z1;</p><p>        z1 = b1 * x - a1 * y + z2;</p><p>        z2 = b2 * x - a2 * y;</p><p>        return y;</p><p>    }</p><p> </p><p>private:</p><p>    float b0, b1, b2, a1, a2;</p><p>    float z1 = 0.0f;</p><p>    float z2 = 0.0f;</p><p>};</p></td></tr></table>

多级 IIR cascade：

<table><tr><td><p>class IIRFilter {</p><p>public:</p><p>    float process_sample(float x) {</p><p>        float y = x;</p><p>        for (auto&amp; section : sections) {</p><p>            y = section.process_sample(y);</p><p>        }</p><p>        return y;</p><p>    }</p><p> </p><p>private:</p><p>    std::array&lt;BiquadFilter, NUM_SECTIONS&gt; sections;</p><p>};</p></td></tr></table>

这样每个样本进来只产生一个输出值，不需要创建输出 vector。

### **1.3 A/C/Z 计权不生成整段输出**

不要这样：

<table><tr><td><p>输入 vector</p><p>↓</p><p>A-weighted vector</p><p>↓</p><p>统计 A-weighted vector</p></td></tr></table>

改成：

<table><tr><td><p>for (size_t i = 0; i &lt; count; ++i) {</p><p>    float x = convert_to_float(samples[i]);</p><p> </p><p>    float xa = a_weighting.process_sample(x);</p><p>    float xc = c_weighting.process_sample(x);</p><p>    float xz = x;</p><p> </p><p>    accumulator.update(xa, xc, xz);</p><p>}</p></td></tr></table>

其中 accumulator.update() 直接更新：

<table><tr><td><p>A-weighted energy</p><p>C-weighted energy</p><p>Z-weighted energy</p><p>C peak</p><p>Z peak</p><p>crest factor</p><p>kurtosis moments</p><p>clipping flags</p></td></tr></table>

### **1.4 每秒只输出统计结果，不输出波形**

每秒生成：

<table><tr><td><p>struct SecondMetrics {</p><p>    float laeq_1s;</p><p>    float lceq_1s;</p><p>    float lzeq_1s;</p><p> </p><p>    float lcpeak_1s;</p><p>    float lzpeak_1s;</p><p> </p><p>    float crest_factor;</p><p>    float kurtosis_1s;</p><p> </p><p>    float dose_niosh_increment;</p><p>    float dose_osha_increment;</p><p> </p><p>    uint32_t flags;</p><p>};</p></td></tr></table>

而不是保存：

<table><tr><td><p>std::vector&lt;float&gt; a_weighted_signal_1s;</p><p>std::vector&lt;float&gt; c_weighted_signal_1s;</p></td></tr></table>

## **阶段 2：重构倍频带/1/3 倍频程分析**

文件中指出，process\_segment 内的 9 个倍频带分支会重复构造输入 vector、bandpass filter 和输出 vector，堆占用随频带次数叠加。

这部分需要单独优化。

### **2.1 不要每个频带复制输入**

错误模式：

<table><tr><td><p>for each band:</p><p>    copy input to vector</p><p>    filter.process(vector)</p><p>    get output vector</p><p>    calculate band Leq</p></td></tr></table>

正确模式：

<table><tr><td><p>for each sample:</p><p>    for each band:</p><p>        y = band_filter[band].process_sample(x);</p><p>        band_accumulator[band].update(y);</p></td></tr></table>

也就是说，每个频带只保存：

<table><tr><td><p>滤波器状态</p><p>能量累计</p><p>峰值或其他统计量</p></td></tr></table>

不保存整段输出。

### **2.2 倍频带分析不一定必须与核心剂量指标同优先级**

对于个人噪声剂量计，指标可以分成两类。

### **第一优先级：必须实时、不能丢**

<table><tr><td><p>LAeq,1s</p><p>LCeq,1s</p><p>LCpeak</p><p>LZpeak</p><p>Dose</p><p>TWA</p><p>LEX,8h</p><p>kurtosis</p><p>crest factor</p><p>overload flag</p></td></tr></table>

### **第二优先级：可选或降频**

<table><tr><td><p>1/1 octave</p><p>1/3 octave</p><p>频谱摘要</p><p>事件声音识别特征</p></td></tr></table>

如果 CPU 预算不足，倍频带分析可以：

1. 只在研究模式开启；

2. 每秒计算一次，而不是每个小段都完整输出；

3. 降低频带数量；

4. 放到低优先级线程；

5. 只对触发事件片段计算；

6. 后处理到 PC/App 端完成。

对第一版硬件，我建议：

<table><tr><td><p>默认固件：核心剂量 + 峰度 + peak</p><p>研究固件：增加 1/3 octave</p><p>PC 工具：完整频谱与高级分析</p></td></tr></table>

# **四、推荐的最终嵌入式算法结构**

## **4.1 数据流结构**

<table><tr><td><p>I2S/PDM DMA</p><p>↓</p><p>10–20 ms ping-pong buffer</p><p>↓</p><p>process_block(pointer, length)</p><p>↓</p><p>逐样本转换为 float</p><p>↓</p><p>A/C/Z streaming filters</p><p>↓</p><p>running accumulators</p><p>↓</p><p>每 1 秒输出 SecondMetrics</p><p>↓</p><p>60 秒 rolling kurtosis</p><p>↓</p><p>8 小时 Dose / LEX,8h</p><p>↓</p><p>外部存储 + BLE/App</p></td></tr></table>

## **4.2 内存结构**

<table><tr><td><p>DMA buffer：固定数组</p><p>滤波器状态：固定数组</p><p>频带滤波器：固定数组</p><p>每秒统计：struct</p><p>60 秒窗口：ring buffer</p><p>长期剂量：累计变量</p></td></tr></table>

禁止：

<table><tr><td><p>std::vector 临时对象</p><p>std::map</p><p>std::string 热路径操作</p><p>new / delete</p><p>malloc / free</p><p>push_back</p><p>filter.process 返回 vector</p></td></tr></table>

允许：

<table><tr><td><p>std::array</p><p>constexpr</p><p>enum class</p><p>固定长度 ring buffer</p><p>指针 + 长度</p><p>状态机</p></td></tr></table>

# **五、具体代码层面的改法**

## **5.1 当前可能的问题写法**

<table><tr><td><p>std::vector&lt;float&gt; input(buffer_start, buffer_end);</p><p> </p><p>auto a = apply_a_weighting(input);</p><p>auto c = apply_c_weighting(input);</p><p> </p><p>auto result = filter.process(input);</p></td></tr></table>

这会导致：

<table><tr><td><p>输入复制一次</p><p>A 计权输出一次</p><p>C 计权输出一次</p><p>滤波输出一次</p><p>倍频带重复多次</p></td></tr></table>

## **5.2 建议改成**

<table><tr><td><p>void NoiseProcessor::process_block(const int32_t* samples, size_t n) {</p><p>    for (size_t i = 0; i &lt; n; ++i) {</p><p>        float x = convert_sample_to_pa(samples[i]);</p><p> </p><p>        float xa = a_filter_.process_sample(x);</p><p>        float xc = c_filter_.process_sample(x);</p><p>        float xz = x;</p><p> </p><p>        second_acc_.update(xa, xc, xz);</p><p> </p><p>#if ENABLE_OCTAVE_BANDS</p><p>        for (size_t b = 0; b &lt; NUM_BANDS; ++b) {</p><p>            float yb = band_filters_[b].process_sample(x);</p><p>            band_acc_[b].update(yb);</p><p>        }</p><p>#endif</p><p> </p><p>        if (second_acc_.sample_count() == SAMPLE_RATE) {</p><p>            SecondMetrics m = second_acc_.finalize();</p><p>            metrics_ring_.push(m);</p><p>            second_acc_.reset();</p><p>        }</p><p>    }</p><p>}</p></td></tr></table>

这个结构有几个好处：

1. 不复制输入；

2. 不生成输出 vector；

3. 不使用 heap；

4. 每个样本只处理一次；

5. A/C/Z/频带结果同步更新；

6. 每秒输出结构化结果；

7. 适合 48 kHz 实时处理。

# **九、建议给工程师的明确修改任务清单**

## **第一优先级：必须改**

1. 替换所有 M\_PI 为 constexpr float PI\_F。

2. 替换实时路径中的 double 常量和函数为 float 版本。

3. process\_segment 禁止构造 std::vector<float>(buffer\_start, buffer\_end)。

4. IIRFilter::process / BiquadFilter::process 改为 process\_sample()。

5. A/C 计权不再返回 vector，直接更新统计量。

6. 所有滤波器状态改为类成员固定数组。

7. 倍频带分析默认关闭，待核心路径稳定后再恢复。

8. 加入运行时间 profiling。

9. 加入 heap allocation 统计。

10. 明确产品目标：热路径零堆分配。

## **第二优先级：应尽快改**

1. A/C/Z 滤波器系数改为 48 kHz 编译期常量表。

2. Dose profile 改为 enum + constexpr table。

3. 60 秒 kurtosis 使用 S1–S4 rolling synthesis。

4. SecondMetrics 结构体紧凑化，去掉不必要的 64-byte align。

5. 输出每秒指标 ring buffer。

6. 建立 PC double 参考算法与嵌入式 float 算法对比测试。

## **第三优先级：优化和扩展**

1. 恢复倍频带分析，但采用 streaming filter bank。

2. 将 1/3 octave 放到低优先级或研究模式。

3. 优化 biquad cascade 顺序和系数量化。

4. 对 LCpeak、kurtosis、高峰度冲击噪声做专门 stress test。

5. 加入事件触发短音频存储，但写入外部 Flash，不占 SRAM 长缓冲。

# **十、最终技术路线建议**

我建议正式确定以下技术路线：

<table><tr><td><p>采样率：48 kHz</p><p>输入格式：24-bit / 32-bit slot</p><p>处理块：20 ms 初始测试</p><p>音频缓冲：DMA ping-pong 固定数组</p><p>算法模式：streaming, no heap</p><p>浮点策略：样本级 float，长期累计 mixed precision</p><p>滤波器：预计算 constexpr biquad coefficients</p><p>指标输出：每 1 秒 SecondMetrics</p><p>峰度：每秒 S1–S4，60 秒 rolling synthesis</p><p>倍频带：第二阶段 streaming filter bank</p><p>实时要求：处理时间 ≤ block 时间的 50%，目标 ≤ 30%</p><p>内存要求：算法热路径零 malloc/new</p></td></tr></table>

这条路线可以同时满足三个目标：

1. **能在 nRF54L15 上稳定运行**；

2. **不牺牲个人噪声剂量计的核心测量能力**；

3. **为后续峰度修正、复杂噪声风险评估和产品认证留下余量。**

# **十一、总结判断**

这份测试结果说明：**算法移植方向已经走到关键节点。** 当前不是单纯修几个编译错误，而是必须从 PC 批处理算法转向嵌入式流式算法。

最关键的问题是：

<table><tr><td><p>std::vector 临时对象太多</p><p>整段音频反复复制</p><p>滤波器返回整段输出</p><p>倍频带分支重复分配</p><p>heap 峰值随段长增长</p><p>处理时间接近采集时间</p></td></tr></table>

最合理的解决方案是：

<table><tr><td><p>固定 DMA buffer</p><p>逐样本/小块流式处理</p><p>固定数组保存滤波器状态</p><p>直接累计声学统计量</p><p>每秒输出指标</p><p>热路径零堆分配</p><p>倍频带作为第二阶段优化</p></td></tr></table>

一句话结论：

*不要通过继续增大 heap 或缩短测试段长来掩盖问题；应重构算法接口和滤波器实现，把当前“整段 vector 输入/输出”的 PC 算法改成“逐样本 streaming + 固定内存 + 每秒统计输出”的嵌入式实时算法。*
